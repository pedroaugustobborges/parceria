Migration Strategy: MV Health Scraper to API

Objective: Replace the Python cron job coletar-produtividade-mv.py with a direct integration between the hospital's AWS environment and the ParcerIA SaaS (Supabase).

1. Architecture Overview

The transition moves from a client-side scraping model to a server-side integration model.

Current State (Deprecated)

  • Source: HTML Report Viewer (http://mvpepprd.saude.go.gov.br...).
  • Method: Python script parsing HTML DOM elements.
  • Risk: High fragility. Changes in UI break the pipeline. Authentication relies on session cookies or browser emulation.

Future State (Target)

  • Source: AWS Environment (Data Lake or Warehouse managed by IT).
  • Method: Structured Data Transfer (JSON/CSV or SQL Replication).
  • Destination: Supabase (PostgreSQL) produtividade table.
  • Benefit: Stability, schema enforcement, auditability.

2. Database Schema Definition

Based on the file produtividade - produtividade_rows.csv, the destination table in Supabase requires the following SQL structure. Ensure the IT team provides data mapping to these columns.

CREATE TABLE public.produtividade (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    codigo_mv BIGINT NOT NULL,
    nome TEXT NOT NULL,
    especialidade TEXT,
    vinculo TEXT,
    
    -- Metrics (Integer types for counts)
    procedimento INTEGER DEFAULT 0,
    parecer_solicitado INTEGER DEFAULT 0,
    parecer_realizado INTEGER DEFAULT 0,
    cirurgia_realizada INTEGER DEFAULT 0,
    prescricao INTEGER DEFAULT 0,
    evolucao INTEGER DEFAULT 0,
    urgencia INTEGER DEFAULT 0,
    ambulatorio INTEGER DEFAULT 0,
    auxiliar INTEGER DEFAULT 0,
    encaminhamento INTEGER DEFAULT 0,
    folha_objetivo_diario INTEGER DEFAULT 0,
    evolucao_diurna_cti INTEGER DEFAULT 0,
    evolucao_noturna_cti INTEGER DEFAULT 0,
    
    -- Metadata
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    data DATE NOT NULL,
    unidade_hospitalar_id BIGINT
);

-- Index for performance on common queries
CREATE INDEX idx_produtividade_data ON public.produtividade(data);
CREATE INDEX idx_produtividade_codigo_mv ON public.produtividade(codigo_mv);

3. AWS to Supabase Integration Patterns

The IT team will ask how to move data from their AWS environment to Supabase. Present these three patterns:

Pattern Mechanism Pros Cons
Postgres FDW (Foreign Data Wrapper) Supabase connects directly to AWS RDS using the postgres_fdw extension. Real-time access; No ETL code required; Treat AWS tables as local tables. Requires network allow-listing; Performance depends on network latency.
AWS Lambda (Push) AWS Lambda function triggers on new data and calls Supabase API (`supabase-js`). Decoupled architecture; AWS team controls the schedule; secure. Requires maintenance of the Lambda function code.
S3 Import (Batch) IT dumps CSVs to an S3 bucket. A Supabase Edge Function reads and imports the file. Good for large bulk updates; standardizes format. Not real-time; Handling file locking/processing state adds complexity.

4. Technical Q&A Preparation

Prepare for the following questions from the AWS/IT team:

Q1: "How do we authenticate with your database?"

Answer: "Supabase provides a standard PostgreSQL connection string. We can create a dedicated user/role for the AWS pipeline with limited permissions (INSERT/UPDATE only) on the produtividade table to maintain security."

Q2: "What is the primary key strategy? How do we handle duplicates?"

Answer: "The table uses a UUID as the primary key. However, for data integrity, we should define a composite unique constraint, likely on codigo_mv + data. If you send a record that already exists, we can use an UPSERT (ON CONFLICT UPDATE) strategy to refresh the metrics without creating duplicates."

Q3: "Do you need a VPN or VPC Peering?"

Answer: "Supabase runs on AWS (or similar cloud infrastructure depending on region). We can whitelist the IP addresses of your NAT Gateway or Lambda functions. VPC peering is possible but typically requires an Enterprise plan; standard connection via SSL/TLS is usually sufficient for this data sensitivity level."

Q4: "What data types do you expect for the metrics?"

Answer: "We expect integers for all count-based fields (e.g., evolucao, prescricao). The data field must be in ISO 8601 format (YYYY-MM-DD). Null values in metrics should default to 0 to prevent calculation errors in the dashboard."
