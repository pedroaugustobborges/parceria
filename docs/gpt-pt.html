1) O que estamos construindo (ponta a ponta)

Objetivo de negócio: parar de “scrapar” o relatório web da MV e passar a consumir dados oficiais via API da MV Saúde, ingerindo no AWS e sincronizando para o Supabase para alimentar análises + chatbot/RAG do ParcerIA.

Fluxo de dados (conceitual)

  • API da MV entrega métricas de produtividade por profissional/especialidade (normalmente em granularidade diária).
  • Tabela de landing no AWS vira o “system of record” dos dados ingeridos (governança da TI).
  • Job de sincronização move registros curados para o Supabase produtividade com chaves corretas e deduplicação.
  • Supabase serve o SaaS (consultas Postgres, dashboards e indexação RAG).
Por que a TI prefere “AWS primeiro”
  • Fronteira de segurança: credenciais/segredos isolados no AWS (IAM + KMS + VPC).
  • Controle operacional: monitoramento/alertas via CloudWatch, backups e processos padronizados.
  • Retenção do dado bruto: manter payload bruto para reprocessar se o schema do SaaS evoluir.
Ponto crítico: decidir quem “dona” da lógica incremental (watermark, retries, dedupe). Em muitos cenários, TI ingere no AWS; o time do produto sincroniza para o Supabase.

3) Aula AWS: onde essa tabela deve morar

No AWS, “uma tabela” normalmente significa uma tabela dentro de um serviço gerenciado de banco de dados. A escolha ideal depende de latência, volume, governança e se você precisa de SQL com joins.

Melhores opções para este caso

  • Amazon RDS / Aurora (PostgreSQL): escolha “padrão ouro” se você quer SQL tradicional + sync incremental simples.
  • S3 + Athena: ótimo se a TI vai aterrar JSON/CSV no S3 e consultar por batch (analytics barato).
  • Redshift: bom para volumes enormes e perfil fortemente analítico (data warehouse).
Na prática: se você quer sincronização previsível para o Supabase, um Postgres no AWS (RDS/Aurora) costuma ser o caminho mais simples de operar.

O que a TI vai priorizar

  • VPC: sub-redes privadas vs acesso público, NAT, security groups.
  • IAM + Segredos: onde ficam credenciais da API (Secrets Manager), e quem pode descriptografar (KMS).
  • Backup/retenção: PITR, snapshots, retenção do dado bruto para auditoria.
  • Observabilidade: métricas/logs no CloudWatch, alarmes e runbooks.
Termos AWS que você precisa dominar na reunião
  • VPC: limite da rede privada; define quem acessa o banco.
  • Security Group: regras de firewall (origem + porta) controlando acesso ao BD.
  • IAM Role/Policy: permissões para serviços (Lambda, ECS) ler Secrets Manager e acessar recursos.
  • KMS: chaves de criptografia protegendo segredos e dados em repouso.
  • RDS Proxy: pool de conexões para Lambdas falando com Postgres (evita “tempestade de conexões”).

4) Padrões de sincronização: push, pull, CDC

Aqui é onde a maioria dos projetos dá certo ou dá errado. Escolha o padrão compatível com suas restrições e maturidade operacional.

Padrão A — Pull (seu job lê AWS → escreve Supabase)

Como funciona: um job agendado (Lambda/ECS ou seu backend) consulta a tabela no AWS buscando “novos/alterados” desde um watermark e faz upsert no Supabase.
  • Prós: modelo mental simples; você controla transformações; backfill fácil.
  • Contras: precisa de um caminho seguro do job para o BD no AWS e para o Supabase.

Padrão B — Push (job da TI escreve direto no Supabase)

Como funciona: após gravar no AWS, o pipeline da TI também faz upsert no Supabase (API/BD).
  • Prós: menos componentes sob sua responsabilidade; TI pode “dona” do fim a fim.
  • Contras: TI precisa gerenciar credenciais do Supabase com cuidado; acoplamento maior ao schema do SaaS.

Padrão C — CDC (Change Data Capture) / Replicação

Como funciona: ferramentas como AWS DMS “streamam” mudanças do BD do AWS para um destino. Pode ser ótimo, mas adiciona complexidade.
  • Prós: quase em tempo real; robusto se bem configurado.
  • Contras: rede + drift de schema + operação mais complexa; muitas vezes é overkill para dados diários.
O que você deve recomendar por padrão

Se o dado é diário e o volume não é absurdo, recomende Padrão A (Pull) com watermark + upsert. É o mais controlável e o mais fácil de depurar.

5) Aula SQL: chaves, índices, upsert, idempotência

Duas chaves que você precisa entender

  • Chave primária (surrogate): seu id uuid no Supabase. Ótimo internamente, mas não necessariamente para dedupe.
  • Chave única natural: combinação de campos do negócio que define um registro unicamente (usada no UPSERT).
Idempotência: o sync deve ser seguro para rodar várias vezes com o mesmo input sem gerar duplicatas. Em SQL, isso significa INSERT ... ON CONFLICT (...) DO UPDATE usando uma unique constraint.

Índices que importam (performance)

  • (data) para filtros por data
  • (unidade_hospitalar_id, data) para dashboards por unidade
  • (codigo_mv, data) para histórico por profissional
  • Índice único para o alvo do conflito (sua chave natural)
Armadilhas de timezone e coluna “data”
  • data deve ser date (sem timezone). Defina como “dia” é calculado (Brasil vs UTC).
  • Use timestamptz para created_at/updated_at (padrão: armazenar em UTC).

6) Integração Supabase: formas seguras de escrever

Supabase é Postgres com APIs. Você pode gravar dados via:

Opção 1 — PostgREST (API REST do Supabase) com Service Role Key

  • Uso: server-to-server com upsert sem abrir portas do banco.
  • Segurança: Service Role ignora RLS. Guarde apenas no servidor (Lambda/ECS/Edge Function).
  • Operação: excelente para cargas pequenas/médias em batch.

Opção 2 — Conexão direta Postgres (driver/psql) com credenciais do BD

  • Uso: bulk load, COPY, ou transformações SQL pesadas.
  • Segurança: requer rede/credenciais bem controladas.
RLS (Row Level Security): para pipelines de ingestão, normalmente você bypassa RLS (service role) e garante qualidade com constraints + auditoria. Clientes não devem escrever dado de ingestão diretamente.
O que a TI vai perguntar sobre rede do Supabase
  • “O BD do Supabase é acessível publicamente?” → Pode ser via string de conexão; restrinja e use segredos.
  • “Dá para conectar do AWS de forma privada?” → Depende de plano e setup; se não, use escrita por API HTTPS.
  • “Você precisa de acesso inbound ao BD do AWS?” → Não. O job roda dentro do AWS e lê o BD privadamente.

7) Arquitetura recomendada (robusta + realista)

Recomendação padrão: TI ingere a API da MV em AWS Aurora/RDS Postgres (raw + curated), e um job agendado no AWS (Lambda ou ECS) lê incremental e faz upsert no Supabase via PostgREST (Service Role).

Por que isso é robusto

  • Menor privilégio: job com acesso somente leitura ao view/tabela curada no AWS e somente escrita no Supabase.
  • Incremental + retomável: armazenar watermark (último updated_at ou data) numa tabela de estado.
  • Upsert idempotente: unique constraint impede duplicatas; reruns são seguros.
  • Operação clara: CloudWatch logs + alarmes; backfill simples ajustando o watermark.
Raw vs curated (o que você deve pedir para a TI criar)
  • raw_mv_produtividade: payload bruto (JSON), timestamps de ingestão, metadados da requisição.
  • mv_produtividade_curated: colunas tipadas alinhadas ao destino, com chave única e source_updated_at.
  • View para sync: view expondo apenas campos necessários e suportando leitura incremental.

8) Perguntas que a TI vai fazer (e como responder)

P1) “Qual é o identificador único? Como vamos deduplicar?”

Vamos definir uma chave única natural no grão diário. Proposta: (unidade_hospitalar_id, data, codigo_mv, especialidade, vinculo). Se a API da MV garantir unicidade sem especialidade/vínculo, simplificamos. Vamos impor isso com unique constraint no Supabase e usar UPSERT.

P2) “Você quer recarga completa ou incremental?”

Incremental. Usar watermark: source_updated_at (preferível) ou uma combinação de data + tempo de ingestão. Manter sync_state com last_successful_watermark. Backfill via “rewind” do watermark.

P3) “Qual frequência você precisa?”

Começar diário (ex.: 02:00 horário local) a menos que o produto exija near real-time. Depois podemos ir para horário. O pipeline deve ser idempotente e seguro para rerun.

P4) “Como o AWS vai escrever no Supabase com segurança?”

Preferível: usar API REST do Supabase (PostgREST) com Service Role Key no AWS Secrets Manager. Job roda no AWS (Lambda/ECS) e escreve no Supabase via HTTPS. Sem exposição no client.

P5) “O Supabase precisa acessar o AWS?”

Não, se o job rodar dentro do AWS. O job lê o BD no AWS de forma privada e escreve no Supabase por HTTPS. Isso evita abrir inbound para o BD no AWS.

P6) “E RLS/auditoria?”

Ingestão usa Service Role (bypassa RLS). Garantimos qualidade com constraints e, se necessário, adicionamos: source_system, ingested_by, ingested_at ou uma tabela de auditoria.

P7) “Qual volume e restrição de performance?”

São agregados diários por profissional, então o volume tende a ser administrável. Vamos fazer upserts em batch (ex.: 500–2.000 linhas por request) e indexar por data e unidade.

P8) “Como lidar com mudança de schema na API da MV?”

Guardar payload bruto (tabela raw ou S3) para reprocessar. A tabela/view curada é o contrato estável. Novas métricas entram como novas colunas opcionais no Supabase com migração controlada.

9) Checklist prático para kickoff & go-live

Contrato de dados

  • Confirmar granularidade e chave de dedupe (diário? por profissional? por especialidade?).
  • Confirmar mapeamento para unidade_hospitalar_id (como a MV identifica a unidade).
  • Definir regra de timezone para data.

Segurança

  • Credenciais da MV no AWS Secrets Manager + KMS.
  • Service Role do Supabase no Secrets Manager (somente servidor).
  • Usuário do BD no AWS somente leitura para tabela/view curada.

Confiabilidade

  • Retries com backoff exponencial para MV API e escrita no Supabase.
  • Estratégia de “dead-letter” (batches com falha logados para replay).
  • Alarmes no CloudWatch para falhas e anomalia de “sem dados ingeridos”.

Operação

  • Procedimento de backfill (rewind do watermark + reprocessamento).
  • Runbook: “o que fazer quando a MV API cair?”
  • Dashboard: contagem diária ingerida; métricas de lag do sync.
Falha comum: não ter unique constraint + sync “só insert” = duplicatas silenciosas. Sempre imponha unicidade e use upsert.

Tabela mínima de “sync_state” (watermark)

CREATE TABLE IF NOT EXISTS public.sync_state (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  pipeline text NOT NULL UNIQUE,             -- ex.: 'mv_produtividade'
  last_watermark timestamptz NOT NULL,       -- último source_updated_at bem-sucedido
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- Inicializar uma vez:
INSERT INTO public.sync_state (pipeline, last_watermark)
VALUES ('mv_produtividade', '2000-01-01T00:00:00Z')
ON CONFLICT (pipeline) DO NOTHING;
Como explicar isso para a TI em 1 minuto

“AWS é a fonte da verdade. O job lê apenas o que mudou desde um watermark e faz upsert no Supabase usando uma chave natural única. Isso deixa o pipeline retomável, tolerante a falhas e fácil de backfilling.”

Feito para a migração da produtividade (MV): MV API → AWS → Supabase. Mantenha simples: chaves fortes, upsert, watermark e bom monitoramento.
