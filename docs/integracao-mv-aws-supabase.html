3. Fundamentos AWS

3.1 Serviços Relevantes para Este Projeto

Amazon RDS (Relational Database Service)

Banco de dados relacional gerenciado. É a opção mais comum para tabelas SQL na AWS.

Por que usar RDS?
  • Compatível com PostgreSQL (mesmo engine do Supabase)
  • Backups automáticos
  • Multi-AZ para alta disponibilidade
  • Escalabilidade vertical (aumentar instância)
-- Exemplo: Criar tabela staging no RDS PostgreSQL
CREATE TABLE staging_produtividade (
    id SERIAL PRIMARY KEY,
    data_extracao TIMESTAMP DEFAULT NOW(),
    payload JSONB NOT NULL,
    processado BOOLEAN DEFAULT FALSE,
    erro TEXT
);

-- Índice para queries de processamento
CREATE INDEX idx_staging_processado
ON staging_produtividade(processado)
WHERE processado = FALSE;

Amazon Aurora

Versão turbinada do RDS, compatível com PostgreSQL mas até 5x mais rápido.

  • Ideal para cargas de trabalho intensivas
  • Serverless v2: escala automaticamente
  • Custo maior, mas melhor performance

AWS Lambda

Funções serverless que executam código sem gerenciar servidores.

# Exemplo: Lambda para sincronizar dados com Supabase
import json
import boto3
import requests
from datetime import datetime

def lambda_handler(event, context):
    # 1. Conectar ao RDS e buscar dados não processados
    # 2. Transformar dados para o schema do Supabase
    # 3. Inserir no Supabase via API REST
    # 4. Marcar como processado no RDS

    supabase_url = "https://xxx.supabase.co/rest/v1/produtividade"
    headers = {
        "apikey": os.environ["SUPABASE_KEY"],
        "Authorization": f"Bearer {os.environ['SUPABASE_KEY']}",
        "Content-Type": "application/json",
        "Prefer": "return=minimal"
    }

    response = requests.post(supabase_url, headers=headers, json=dados)

    return {
        "statusCode": 200,
        "body": json.dumps({"processados": len(dados)})
    }

4. Modelagem SQL

4.1 Schema da Tabela AWS (Staging)

A tabela de staging na AWS deve ser flexível para receber os dados brutos da API MV:

-- Tabela de staging para dados brutos da API MV
CREATE TABLE mv_produtividade_raw (
    -- Identificação
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- Metadados da extração
    extracted_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    source_endpoint TEXT,
    request_params JSONB,

    -- Dados brutos da API
    raw_response JSONB NOT NULL,

    -- Controle de processamento
    processed BOOLEAN DEFAULT FALSE,
    processed_at TIMESTAMP WITH TIME ZONE,
    sync_status TEXT CHECK (sync_status IN ('pending', 'synced', 'error')),
    error_message TEXT,
    retry_count INTEGER DEFAULT 0
);

-- Índices para performance
CREATE INDEX idx_raw_pending ON mv_produtividade_raw(sync_status)
    WHERE sync_status = 'pending';
CREATE INDEX idx_raw_extracted ON mv_produtividade_raw(extracted_at);

4.2 Schema da Tabela Supabase (Destino)

A tabela produtividade no Supabase deve ter um schema bem definido:

-- Tabela final no Supabase (ParcerIA)
CREATE TABLE produtividade (
    -- Chave primária
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- Referências
    unidade_hospitalar_id UUID REFERENCES unidades_hospitalares(id),
    contrato_id UUID REFERENCES contratos(id),
    profissional_id UUID REFERENCES usuarios(id),

    -- Dados de produtividade
    data_referencia DATE NOT NULL,
    tipo_atendimento TEXT,
    quantidade_atendimentos INTEGER DEFAULT 0,
    horas_trabalhadas DECIMAL(5,2),
    valor_total DECIMAL(12,2),

    -- Metadados MV
    mv_id TEXT,  -- ID original do MV para deduplicação
    mv_codigo_procedimento TEXT,

    -- Auditoria
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    source TEXT DEFAULT 'mv_api',

    -- Constraint para evitar duplicatas
    UNIQUE(mv_id)
);

-- Índices para queries do dashboard
CREATE INDEX idx_prod_data ON produtividade(data_referencia);
CREATE INDEX idx_prod_unidade ON produtividade(unidade_hospitalar_id);
CREATE INDEX idx_prod_contrato ON produtividade(contrato_id);

-- Trigger para updated_at
CREATE TRIGGER update_produtividade_timestamp
    BEFORE UPDATE ON produtividade
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at_column();

-- Row Level Security
ALTER TABLE produtividade ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Admins podem ver tudo" ON produtividade
    FOR SELECT
    USING (
        auth.jwt() ->> 'tipo' IN ('administrador-agir-corporativo', 'administrador-agir-planta')
    );

CREATE POLICY "Parceiros veem apenas seus dados" ON produtividade
    FOR SELECT
    USING (
        contrato_id IN (
            SELECT contrato_id FROM usuario_contrato
            WHERE usuario_id = auth.uid()
        )
    );

4.3 Query de Transformação (ETL)

-- Query para transformar dados do staging para o formato final
INSERT INTO produtividade (
    unidade_hospitalar_id,
    contrato_id,
    profissional_id,
    data_referencia,
    tipo_atendimento,
    quantidade_atendimentos,
    horas_trabalhadas,
    valor_total,
    mv_id,
    mv_codigo_procedimento
)
SELECT
    -- Mapear código MV para UUID da unidade
    (SELECT id FROM unidades_hospitalares
     WHERE codigo_mv = raw.raw_response->>'codigo_unidade') AS unidade_hospitalar_id,

    -- Mapear contrato (pode precisar de lógica adicional)
    (SELECT id FROM contratos
     WHERE numero_contrato = raw.raw_response->>'contrato') AS contrato_id,

    -- Mapear profissional por CPF
    (SELECT id FROM usuarios
     WHERE cpf = raw.raw_response->>'cpf_profissional') AS profissional_id,

    -- Dados diretos
    (raw.raw_response->>'data_atendimento')::DATE,
    raw.raw_response->>'tipo_atendimento',
    (raw.raw_response->>'qtd_atendimentos')::INTEGER,
    (raw.raw_response->>'horas')::DECIMAL,
    (raw.raw_response->>'valor')::DECIMAL,
    raw.raw_response->>'id_mv',
    raw.raw_response->>'codigo_procedimento'

FROM mv_produtividade_raw raw
WHERE raw.sync_status = 'pending'
ON CONFLICT (mv_id) DO UPDATE SET
    quantidade_atendimentos = EXCLUDED.quantidade_atendimentos,
    horas_trabalhadas = EXCLUDED.horas_trabalhadas,
    valor_total = EXCLUDED.valor_total,
    updated_at = NOW();

5. Integração com Supabase

5.1 O que é Supabase?

Supabase é uma plataforma Backend-as-a-Service (BaaS) open source que oferece:

PostgreSQL Gerenciado

Banco de dados PostgreSQL completo com extensões (PostGIS, pg_vector, etc).

API REST Automática

PostgREST gera endpoints REST automaticamente para todas as tabelas.

Realtime

WebSockets para atualizações em tempo real via PostgreSQL LISTEN/NOTIFY.

Auth Integrado

Sistema de autenticação com JWT, OAuth, Magic Links, etc.

5.2 Métodos de Integração AWS → Supabase

Opção A: API REST (Recomendado)

Usar a API REST do Supabase para inserir dados. Mais simples e seguro.

# Python: Inserir dados via API REST do Supabase
import requests
import os

SUPABASE_URL = os.environ["SUPABASE_URL"]  # https://xxx.supabase.co
SUPABASE_KEY = os.environ["SUPABASE_SERVICE_KEY"]  # service_role key para bypass RLS

def sync_to_supabase(records: list[dict]):
    """Sincroniza registros para a tabela produtividade no Supabase"""

    url = f"{SUPABASE_URL}/rest/v1/produtividade"

    headers = {
        "apikey": SUPABASE_KEY,
        "Authorization": f"Bearer {SUPABASE_KEY}",
        "Content-Type": "application/json",
        "Prefer": "resolution=merge-duplicates",  # Upsert
    }

    # Batch insert (max 1000 por request)
    batch_size = 1000
    for i in range(0, len(records), batch_size):
        batch = records[i:i + batch_size]
        response = requests.post(url, headers=headers, json=batch)

        if response.status_code not in [200, 201]:
            raise Exception(f"Erro Supabase: {response.text}")

    return len(records)

# Exemplo de uso
dados = [
    {
        "data_referencia": "2024-01-15",
        "unidade_hospitalar_id": "uuid-da-unidade",
        "quantidade_atendimentos": 42,
        "mv_id": "MV-123456"
    }
]
sync_to_supabase(dados)

Opção B: Conexão Direta PostgreSQL

Conectar diretamente ao PostgreSQL do Supabase (requer IP allowlist).

# Python: Conexão direta ao PostgreSQL do Supabase
import psycopg2
from psycopg2.extras import execute_values

# Connection string do Supabase (Settings > Database)
DATABASE_URL = "postgresql://postgres.[project-ref]:[password]@aws-0-us-east-1.pooler.supabase.com:6543/postgres"

def bulk_insert_direct(records: list[tuple]):
    """Inserção direta via PostgreSQL - mais rápido para grandes volumes"""

    conn = psycopg2.connect(DATABASE_URL)
    try:
        with conn.cursor() as cur:
            execute_values(
                cur,
                """
                INSERT INTO produtividade
                (data_referencia, unidade_hospitalar_id, quantidade_atendimentos, mv_id)
                VALUES %s
                ON CONFLICT (mv_id) DO UPDATE SET
                    quantidade_atendimentos = EXCLUDED.quantidade_atendimentos,
                    updated_at = NOW()
                """,
                records
            )
        conn.commit()
    finally:
        conn.close()
Atenção para Conexão Direta:
  • Requer configurar IP allowlist no Supabase ou usar connection pooling
  • Lambda tem IPs dinâmicos - use NAT Gateway ou VPC para IP fixo
  • Prefira a API REST para simplicidade

Opção C: Supabase Edge Functions + Webhook

AWS envia webhook para Edge Function do Supabase processar.

// Supabase Edge Function (Deno/TypeScript)
// supabase/functions/sync-produtividade/index.ts

import { createClient } from '@supabase/supabase-js'

Deno.serve(async (req) => {
    // Validar secret do webhook
    const authHeader = req.headers.get('x-webhook-secret')
    if (authHeader !== Deno.env.get('WEBHOOK_SECRET')) {
        return new Response('Unauthorized', { status: 401 })
    }

    const supabase = createClient(
        Deno.env.get('SUPABASE_URL')!,
        Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
    )

    const payload = await req.json()

    const { error } = await supabase
        .from('produtividade')
        .upsert(payload.records, { onConflict: 'mv_id' })

    if (error) {
        return new Response(JSON.stringify({ error }), { status: 500 })
    }

    return new Response(JSON.stringify({ success: true }))
})

6. Segurança e Boas Práticas

6.1 Gestão de Credenciais

AWS Secrets Manager

# Buscar secret no Lambda
import boto3
import json

def get_secret(secret_name):
    client = boto3.client('secretsmanager')
    response = client.get_secret_value(
        SecretId=secret_name
    )
    return json.loads(response['SecretString'])

# Uso
secrets = get_secret('parceria/supabase')
supabase_key = secrets['service_key']

Supabase Vault

-- Armazenar secret no Vault
SELECT vault.create_secret(
    'mv_api_key',
    'chave-secreta-da-api-mv'
);

-- Usar em função
CREATE FUNCTION get_mv_key()
RETURNS TEXT AS $$
    SELECT decrypted_secret
    FROM vault.decrypted_secrets
    WHERE name = 'mv_api_key';
$$ LANGUAGE SQL SECURITY DEFINER;

6.2 Princípios de Segurança

NUNCA faça isso:
  • Hardcode de credenciais no código fonte
  • Commitar arquivos .env no Git
  • Usar a anon key para operações server-side
  • Expor connection strings em logs
  • Desabilitar RLS "para facilitar"
SEMPRE faça isso:
  • Use service_role key apenas em backends seguros
  • Implemente RLS em todas as tabelas com dados sensíveis
  • Rotacione credenciais periodicamente
  • Use IAM roles em vez de access keys na AWS
  • Mantenha logs de auditoria

7. Perguntas Técnicas Frequentes

P: Como garantir que não haverá duplicatas?

R: Usar o campo mv_id (ID original do MV) como identificador único. Implementar constraint UNIQUE e usar UPSERT (ON CONFLICT DO UPDATE) nas inserções. A API REST do Supabase suporta isso via header Prefer: resolution=merge-duplicates.

P: Qual a frequência de sincronização recomendada?

R: Depende do volume e requisitos de negócio. Para produtividade médica, sincronização a cada 1-2 horas geralmente é suficiente. Se precisar de dados mais frescos, pode-se usar near-realtime (5-15 min). Considerar horários de pico do sistema MV.

P: Como lidar com falhas na sincronização?

R: Implementar retry com backoff exponencial. Usar Dead Letter Queue (DLQ) para mensagens que falharem após N tentativas. Manter campo sync_status e retry_count na tabela staging. Configurar alertas no CloudWatch para falhas.

P: Como a AWS vai conectar no Supabase se o IP muda?

R: Duas opções:

  1. Recomendado: Usar API REST do Supabase (não precisa de IP fixo)
  2. Se precisar conexão direta: Lambda em VPC com NAT Gateway (IP fixo) ou usar o Connection Pooler do Supabase com SSL

P: Quem vai criar a tabela na AWS?

R: O time de TI/Infraestrutura cria a tabela no RDS. Você deve fornecer:

  • Schema SQL desejado (DDL)
  • Requisitos de índices
  • Estimativa de volume de dados
  • Requisitos de retenção

P: E se a API do MV estiver fora do ar?

R: O sistema deve ser resiliente:

  • Retry automático com backoff
  • Circuit breaker para evitar sobrecarregar API em recuperação
  • Alerta para equipe quando falhar por muito tempo
  • Os dados do Supabase continuam disponíveis (última sincronização)

P: Como validar que os dados estão corretos?

R: Implementar verificações:

  • Comparar totais: soma de atendimentos AWS vs Supabase
  • Verificar ranges: datas dentro do período esperado
  • Alertar anomalias: variação > 20% do dia anterior
  • Reconciliação periódica com relatório original do MV

P: Qual o custo estimado dessa arquitetura?

R: Estimativa mensal para volume médio:
