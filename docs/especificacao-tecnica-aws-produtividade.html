1. Contexto do Projeto

1.1 Situação Atual

Atualmente, os dados de produtividade médica são coletados via web scraping do relatório MV:

  • URL: http://mvpepprd.saude.go.gov.br/report-executor/report-viewer?id=7076
  • Script: coletar-produtividade-mv.py
  • Execução: Cron diário às 2h da manhã
  • Destino: Tabela produtividade no Supabase (ParcerIA)

1.2 Nova Arquitetura

Com a disponibilização da API REST do MV Saúde, a arquitetura será:

┌─────────────┐      ┌─────────────────┐      ┌─────────────────┐
│  MV Saúde   │      │   AWS RDS       │      │    Supabase     │
│    API      │ ───► │  (Staging)      │ ───► │  produtividade  │
└─────────────┘      └─────────────────┘      └─────────────────┘
                            │
                     Lambda/EventBridge
                      (ETL Scheduler)
Objetivo: O time de TI cria a tabela de staging no RDS. O time ParcerIA desenvolve o ETL para sincronizar com o Supabase.

2. Schema SQL (DDL)

2.1 Tabela de Staging (AWS RDS)

Esta tabela recebe os dados brutos da API MV e serve como buffer antes da sincronização com o Supabase.

-- ==============================================================================
-- TABELA DE STAGING: mv_produtividade_staging
-- Descrição: Recebe dados brutos da API MV Saúde para posterior sincronização
-- Engine: PostgreSQL 15+ (RDS)
-- ==============================================================================

CREATE TABLE mv_produtividade_staging (
    -- === IDENTIFICAÇÃO ===
    id                      UUID            PRIMARY KEY DEFAULT gen_random_uuid(),

    -- === DADOS DO PROFISSIONAL (da API MV) ===
    codigo_mv               VARCHAR(20)     NOT NULL,      -- Código do prestador no MV
    nome                    VARCHAR(255)    NOT NULL,      -- Nome do profissional
    especialidade           VARCHAR(100),                   -- Especialidade médica
    vinculo                 VARCHAR(100),                   -- Tipo de vínculo

    -- === DATA DE REFERÊNCIA ===
    data_referencia         DATE            NOT NULL,      -- Data dos atendimentos

    -- === MÉTRICAS DE PRODUTIVIDADE ===
    procedimento            INTEGER         DEFAULT 0,     -- Qtd procedimentos realizados
    parecer_solicitado      INTEGER         DEFAULT 0,     -- Qtd pareceres solicitados
    parecer_realizado       INTEGER         DEFAULT 0,     -- Qtd pareceres realizados
    cirurgia_realizada      INTEGER         DEFAULT 0,     -- Qtd cirurgias realizadas
    prescricao              INTEGER         DEFAULT 0,     -- Qtd prescrições
    evolucao                INTEGER         DEFAULT 0,     -- Qtd evoluções
    urgencia                INTEGER         DEFAULT 0,     -- Qtd atendimentos urgência
    ambulatorio             INTEGER         DEFAULT 0,     -- Qtd atendimentos ambulatório
    auxiliar                INTEGER         DEFAULT 0,     -- Qtd como médico auxiliar
    encaminhamento          INTEGER         DEFAULT 0,     -- Qtd encaminhamentos
    folha_objetivo_diario   INTEGER         DEFAULT 0,     -- Qtd folhas objetivo diário
    evolucao_diurna_cti     INTEGER         DEFAULT 0,     -- Qtd evoluções diurnas CTI
    evolucao_noturna_cti    INTEGER         DEFAULT 0,     -- Qtd evoluções noturnas CTI

    -- === DADOS OPCIONAIS DA API ===
    unidade_codigo          VARCHAR(20),                   -- Código da unidade no MV
    raw_payload             JSONB,                         -- Payload original (debug)

    -- === CONTROLE DE SINCRONIZAÇÃO ===
    sync_status             VARCHAR(20)     DEFAULT 'pending'
                            CHECK (sync_status IN ('pending', 'synced', 'error', 'skipped')),
    sync_attempts           INTEGER         DEFAULT 0,
    sync_error_message      TEXT,
    synced_at               TIMESTAMP WITH TIME ZONE,

    -- === AUDITORIA ===
    source_api_version      VARCHAR(20),                   -- Versão da API MV
    extracted_at            TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    created_at              TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at              TIMESTAMP WITH TIME ZONE DEFAULT NOW(),

    -- === CONSTRAINTS ===
    CONSTRAINT uk_staging_codigo_data UNIQUE (codigo_mv, data_referencia)
);

-- Comentários da tabela
COMMENT ON TABLE mv_produtividade_staging IS 'Staging table for MV Saúde productivity data before sync to Supabase';
COMMENT ON COLUMN mv_produtividade_staging.codigo_mv IS 'Unique provider code in MV system';
COMMENT ON COLUMN mv_produtividade_staging.sync_status IS 'pending=awaiting sync, synced=sent to Supabase, error=failed, skipped=ignored';

3. Requisitos de Índices

3.1 Índices Recomendados

-- ==============================================================================
-- ÍNDICES PARA PERFORMANCE
-- ==============================================================================

-- 1. Índice para busca de registros pendentes (ETL)
--    Query: SELECT * FROM mv_produtividade_staging WHERE sync_status = 'pending'
--    Uso: Alta frequência (a cada execução do sync job)
CREATE INDEX idx_staging_sync_pending
ON mv_produtividade_staging(sync_status)
WHERE sync_status = 'pending';

-- 2. Índice para busca por data (relatórios e filtros)
--    Query: SELECT * FROM mv_produtividade_staging WHERE data_referencia BETWEEN x AND y
--    Uso: Moderada (consultas de relatório)
CREATE INDEX idx_staging_data_referencia
ON mv_produtividade_staging(data_referencia DESC);

-- 3. Índice para busca por profissional
--    Query: SELECT * FROM mv_produtividade_staging WHERE codigo_mv = ?
--    Uso: Moderada (debug, verificação individual)
CREATE INDEX idx_staging_codigo_mv
ON mv_produtividade_staging(codigo_mv);

-- 4. Índice para registros com erro (monitoramento)
--    Query: SELECT * FROM mv_produtividade_staging WHERE sync_status = 'error'
--    Uso: Baixa (alertas e troubleshooting)
CREATE INDEX idx_staging_sync_errors
ON mv_produtividade_staging(sync_status, sync_attempts)
WHERE sync_status = 'error';

-- 5. Índice para cleanup de registros antigos sincronizados
--    Query: DELETE FROM mv_produtividade_staging WHERE synced_at < ? AND sync_status = 'synced'
--    Uso: Baixa (job de limpeza semanal/mensal)
CREATE INDEX idx_staging_synced_cleanup
ON mv_produtividade_staging(synced_at)
WHERE sync_status = 'synced';

3.2 Justificativa dos Índices

Índice Tipo Frequência de Uso Justificativa
idx_staging_sync_pending Parcial (WHERE) Alta (hourly) ETL busca registros pendentes a cada execução. Índice parcial é mais eficiente.
idx_staging_data_referencia B-tree DESC Moderada Consultas de relatório geralmente buscam dados recentes primeiro.
idx_staging_codigo_mv B-tree Moderada Verificação de dados específicos de um profissional.
idx_staging_sync_errors Composto + Parcial Baixa Monitoramento de falhas e retry automático.
idx_staging_synced_cleanup Parcial (WHERE) Baixa Job de limpeza de registros já sincronizados.
Nota sobre o UNIQUE constraint: A constraint uk_staging_codigo_data já cria um índice implícito para (codigo_mv, data_referencia), usado para deduplicação (UPSERT).

5.2 Job de Limpeza Automática

-- ==============================================================================
-- PROCEDURE DE LIMPEZA AUTOMÁTICA
-- Executar: Semanalmente (domingo 3h AM)
-- ==============================================================================

CREATE OR REPLACE PROCEDURE cleanup_staging_old_records()
LANGUAGE plpgsql
AS $$
DECLARE
    deleted_synced INTEGER;
    deleted_errors INTEGER;
BEGIN
    -- 1. Remover registros sincronizados há mais de 30 dias
    DELETE FROM mv_produtividade_staging
    WHERE sync_status = 'synced'
      AND synced_at < NOW() - INTERVAL '30 days';

    GET DIAGNOSTICS deleted_synced = ROW_COUNT;

    -- 2. Remover registros com erro há mais de 90 dias
    DELETE FROM mv_produtividade_staging
    WHERE sync_status = 'error'
      AND created_at < NOW() - INTERVAL '90 days';

    GET DIAGNOSTICS deleted_errors = ROW_COUNT;

    -- 3. Log da operação
    RAISE NOTICE 'Cleanup completed: % synced, % errors deleted',
        deleted_synced, deleted_errors;
END;
$$;

6. Triggers e Funções Auxiliares

-- ==============================================================================
-- TRIGGER PARA ATUALIZAÇÃO AUTOMÁTICA DE updated_at
-- ==============================================================================

CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_staging_updated_at
    BEFORE UPDATE ON mv_produtividade_staging
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at_column();

-- ==============================================================================
-- FUNÇÃO PARA MARCAR COMO SINCRONIZADO
-- ==============================================================================

CREATE OR REPLACE FUNCTION mark_as_synced(record_ids UUID[])
RETURNS INTEGER AS $$
DECLARE
    affected INTEGER;
BEGIN
    UPDATE mv_produtividade_staging
    SET
        sync_status = 'synced',
        synced_at = NOW()
    WHERE id = ANY(record_ids)
      AND sync_status = 'pending';

    GET DIAGNOSTICS affected = ROW_COUNT;
    RETURN affected;
END;
$$ LANGUAGE plpgsql;

Exemplo em Python (para Lambda):

import requests
import os

def sync_to_supabase(records: list[dict]) -> int:
    """
    Envia registros para a tabela produtividade no Supabase.

    Args:
        records: Lista de dicionários com os dados de produtividade

    Returns:
        Quantidade de registros processados
    """

    # Credenciais do Secrets Manager
    SUPABASE_URL = os.environ["SUPABASE_URL"]
    SUPABASE_KEY = os.environ["SUPABASE_SERVICE_KEY"]

    url = f"{SUPABASE_URL}/rest/v1/produtividade"

    headers = {
        "apikey": SUPABASE_KEY,
        "Authorization": f"Bearer {SUPABASE_KEY}",
        "Content-Type": "application/json",
        "Prefer": "resolution=merge-duplicates",  # UPSERT
    }

    # Supabase aceita até 1000 registros por requisição
    batch_size = 1000
    total_synced = 0

    for i in range(0, len(records), batch_size):
        batch = records[i:i + batch_size]

        response = requests.post(
            url,
            headers=headers,
            json=batch,
            timeout=30
        )

        if response.status_code not in [200, 201]:
            raise Exception(
                f"Erro Supabase ({response.status_code}): {response.text}"
            )

        total_synced += len(batch)

    return total_synced

Importante para Conexão Direta:
  • Use o Connection Pooler (porta 6543), não a conexão direta (5432)
  • Lambda tem IPs dinâmicos - o pooler resolve isso automaticamente
  • Se usar conexão direta, precisará de NAT Gateway para IP fixo
  • SSL é obrigatório: sslmode=require

8.4 Tipos de Chaves do Supabase (Importante!)

O Supabase possui diferentes chaves com diferentes níveis de acesso:

Chave Uso RLS Segurança
anon (pública) Frontend, apps cliente ✅ Ativo ⚠️ Pode ser exposta no browser
service_role (privada) Backend, Lambda, ETL ❌ Bypass 🔒 NUNCA expor no frontend
🚨 NUNCA faça isso:
  • Usar a service_role key em código frontend/cliente
  • Commitar credentials em repositórios Git
  • Enviar credentials por email ou chat sem criptografia
  • Hardcode de credentials no código Lambda
✅ SEMPRE faça isso:
  • Armazenar credentials no AWS Secrets Manager
  • Usar variáveis de ambiente no Lambda
  • Rotacionar a service_role key periodicamente
  • Monitorar uso via logs do Supabase

import boto3
import json

def get_supabase_credentials():
    """Busca credenciais do Supabase no Secrets Manager."""

    client = boto3.client('secretsmanager')

    response = client.get_secret_value(
        SecretId='parceria/supabase-credentials'
    )

    return json.loads(response['SecretString'])

# Uso:
creds = get_supabase_credentials()

9. Resumo Executivo para TI

O que precisamos:

  1. Criar tabela mv_produtividade_staging no RDS PostgreSQL
  2. Criar índices conforme especificado
  3. Configurar usuário com permissões de INSERT, UPDATE, SELECT, DELETE
  4. Liberar acesso para Lambda (Security Group / VPC)
  5. Fornecer connection string para o time ParcerIA

Características do workload:

  • Volume: Muito baixo (~150 registros/dia)
  • Crescimento: Linear e previsível
  • Padrão: Write-once, read-few (ETL staging)
  • Criticidade: Média (dados podem ser reprocessados)
  • Compliance: Dados de profissionais, não de pacientes

Entregáveis Esperados do Time de TI:

1 Instância RDS PostgreSQL provisionada TI/Infra
2 Tabela e índices criados TI/DBA

5 Connection string entregue (via Secrets Manager) TI/Infra
