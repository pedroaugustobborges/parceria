1) What we are building (end-to-end)

Business goal: stop scraping the MV web report and instead ingest authoritative productivity metrics via MV API into AWS, then synchronize into Supabase to power analytics + the ParcerIA chatbot/RAG.

Data flow (conceptual)

  • MV API delivers daily (or near-real-time) productivity counts per professional and specialty.
  • AWS landing table becomes the “system of record” for raw ingested MV API payloads (owned by IT).
  • Sync job moves curated records into Supabase produtividade with correct keys and deduplication.
  • Supabase serves the SaaS (Postgres queries, dashboards, and RAG indexing).
Why IT prefers “AWS table first”
  • Security boundary: IT can isolate credentials, networking, and API secrets inside AWS (IAM + KMS + VPC).
  • Operational control: monitoring/alerts via CloudWatch, standardized backups, and change management.
  • Raw data retention: they can store raw payloads and reprocess if your SaaS schema changes.
Key alignment point: decide who “owns” the incremental sync logic (watermark, retries, dedupe). In many orgs, IT owns ingestion into AWS; the product team owns sync into Supabase.

3) AWS lesson: where the table should live

In AWS, “a table” typically means a table inside a managed database service. The best choice depends on latency, volume, governance, and whether you need SQL joins.

Best fits for your case

  • Amazon RDS / Aurora (PostgreSQL): best default if you want standard SQL + easy incremental reads.
  • S3 + Athena: best if IT lands raw JSON/CSV in S3 and you query it cheaply (batch analytics).
  • Redshift: best if the dataset is huge and heavily analytical (warehouse patterns).
Reality check: if you want fast, predictable syncing into Supabase, an AWS Postgres table (RDS/Aurora) is usually simplest operationally.

What IT will care about

  • VPC placement: private subnets vs public access, NAT, security groups.
  • IAM + Secrets: where MV API credentials live (Secrets Manager), who can decrypt (KMS).
  • Backup/retention: PITR, snapshots, and raw payload retention for audits.
  • Observability: CloudWatch metrics/logs, alarms, and runbooks.
AWS terms you must be fluent with in the meeting
  • VPC: private network boundary; determines who can reach the database.
  • Security Group: firewall rules (source IP/SG + port) controlling DB access.
  • IAM Role/Policy: permissions for services (Lambda, ECS) to read Secrets Manager or talk to RDS.
  • KMS: encryption keys protecting secrets and data at rest.
  • RDS Proxy: connection pooling for Lambdas hitting Postgres (prevents connection storms).

4) Data sync patterns: push, pull, CDC

This is where most projects succeed or fail. Pick a pattern that matches your constraints and your team’s operational maturity.

Pattern A — Pull (your job reads AWS → writes Supabase)

How it works: a scheduled job (could run in AWS Lambda/ECS or your backend) queries AWS table for “new/changed” rows since a watermark and upserts into Supabase.
  • Pros: simplest mental model; you control transformations; easy backfills.
  • Cons: requires secure network path from the job to both AWS DB and Supabase.

Pattern B — Push (IT job writes Supabase directly)

How it works: IT ingestion pipeline, after writing AWS table, also calls Supabase (PostgREST/DB) to upsert.
  • Pros: fewer moving parts for you; IT can own reliability end-to-end.
  • Cons: requires IT to manage Supabase credentials securely; tighter coupling with your SaaS schema.

Pattern C — CDC (Change Data Capture) / Replication

How it works: tools like AWS DMS stream changes from AWS DB into a target. This can be powerful, but adds complexity.
  • Pros: near-real-time; robust if set up correctly.
  • Cons: networking + schema drift + operational complexity; often overkill for daily productivity metrics.
What you should recommend by default

If your dataset is daily and not massive, recommend Pattern A (Pull) with incremental watermark + upsert. It’s the most controllable and easiest to debug.

5) SQL lesson: keys, indexes, upserts, idempotency

Two keys you must understand

  • Primary key (surrogate): your id uuid in Supabase. Great for internal references, not always good for dedupe.
  • Natural unique key: combination of business fields that defines a record uniquely (used for UPSERT).
Idempotency: the sync must be safe to run multiple times with the same input without creating duplicates. In SQL, that means INSERT ... ON CONFLICT (...) DO UPDATE using a unique constraint.

Indexes that matter (performance)

  • (data) for date filtering
  • (unidade_hospitalar_id, data) for unit dashboards
  • (codigo_mv, data) for professional history
  • Unique index for the upsert conflict target (your natural key)
Timezone & “data” column pitfalls
  • data should be a date (no timezone). Define how “day” is computed (Brazil local vs UTC).
  • Use timestamptz for created_at/updated_at (always UTC storage is standard).

6) Supabase integration: safe ways to write

Supabase is Postgres plus APIs. You can write data either via:

Option 1 — PostgREST (Supabase REST API) with Service Role Key

  • Use case: server-to-server upserts without opening database ports.
  • Security: Service Role bypasses RLS. Keep it ONLY on the server (Lambda/ECS/Edge Function).
  • Operational: easiest for small/medium batch loads.

Option 2 — Direct Postgres connection (psql/driver) using DB credentials

  • Use case: bulk loads, COPY, or heavy SQL transformations.
  • Security: must manage network access and credentials very carefully.
RLS reminder: for ingestion pipelines, you typically bypass RLS (service role) and enforce correctness with constraints + audit logs. Do not let clients write ingestion data directly.
What IT will ask about Supabase networking
  • “Is Supabase DB publicly reachable?” → It can be reachable via connection string; you should restrict exposure and use secrets.
  • “Can we connect from AWS privately?” → Possibly via peering/private networking depending on Supabase plan/provider; otherwise use API-based writes.
  • “Do you need inbound access to AWS DB?” → Only from a controlled job (Lambda/ECS) inside AWS, not from browsers.

7) Recommended architecture (robust + realistic)

Recommended default: IT ingests MV API into AWS Aurora/RDS Postgres (raw + curated tables), then a scheduled AWS job (Lambda or ECS task) pulls incremental changes and upserts into Supabase via PostgREST using Service Role.

Why this is robust

  • Least privilege: the job has read-only access to AWS curated view/table and write-only capability to Supabase target.
  • Incremental + resumable: store a watermark (last updated_at or last data) in a sync_state table.
  • Idempotent upsert: unique constraint prevents duplicates; reruns are safe.
  • Operational clarity: CloudWatch logs + alarms on failures; easy backfills by adjusting watermark.
Raw vs curated tables (what you should ask IT to build)
  • raw_mv_produtividade: store raw API payload (JSON), ingestion timestamps, request metadata.
  • mv_produtividade_curated: clean typed columns matching your destination schema, plus a stable unique key and source_updated_at.
  • View for syncing: a view that exposes only the curated fields you need and supports incremental reads.

8) The questions IT will ask you (and your answers)

Q1) “What is the unique identifier for a record? How do we dedupe?”

We should define a natural unique key at the daily grain. Proposed: (unidade_hospitalar_id, data, codigo_mv, especialidade, vinculo). If MV API guarantees uniqueness without specialty/vínculo, we can simplify. We’ll enforce this with a unique constraint in Supabase and use UPSERT.

Q2) “What update strategy do you want: full reload or incremental?”

Incremental. Use a watermark: either source_updated_at (preferred) or a combination of data + ingestion time. Keep a sync_state row with last_successful_watermark. Support backfill by rewinding it.

Q3) “How often do you need the data?”

Start with daily (e.g., 02:00 local) unless product needs near-real-time. We can move to hourly later. The pipeline should be idempotent and safe for reruns.

Q4) “How will AWS reach Supabase securely?”

Preferred: use Supabase REST (PostgREST) with Service Role Key stored in AWS Secrets Manager. Job runs in AWS (Lambda/ECS), calls Supabase over HTTPS. No client exposure.

Q5) “Do we need network access from Supabase to AWS?”

Not necessary if we run the sync inside AWS. AWS job reads AWS DB privately and writes to Supabase via HTTPS. This avoids inbound holes into AWS DB.

Q6) “What about RLS and audit?”

Ingestion uses Service Role (bypasses RLS). We enforce correctness with DB constraints, and optionally add: source_system, ingested_by, ingested_at fields or a separate audit table.

Q7) “What’s your expected volume and performance constraints?”

The table is daily aggregates per professional, so volume is typically manageable. We’ll batch upserts (e.g., 500–2,000 rows per request) and index by date and unit.

Q8) “How do we handle schema changes from MV API?”

Keep raw payloads (raw table or S3) for reprocessing. Curated table/view is the stable contract. Any new metrics can be added as new nullable columns in Supabase with a controlled migration.

9) Practical checklist for kickoff & go-live

Data contract

  • Confirm record grain and dedupe key (daily? per professional? per specialty?).
  • Confirm mapping for unidade_hospitalar_id (how MV identifies the unit).
  • Define timezone rules for data.

Security

  • MV API creds in AWS Secrets Manager + KMS.
  • Supabase Service Role key in Secrets Manager (server-only).
  • Read-only DB user for curated view/table in AWS.

Reliability

  • Retries with exponential backoff for MV API and Supabase writes.
  • Dead-letter strategy (failed batches logged for replay).
  • CloudWatch alarms on failures and on “no data ingested” anomalies.

Operations

  • Backfill procedure (rewind watermark, re-run).
  • Runbook: “What to do when MV API is down?”
  • Dashboard: ingestion counts per day; sync lag metrics.
Common failure mode: no unique constraint + “insert-only” sync = silent duplicates. Always enforce uniqueness and upsert.

Minimal “sync_state” table (watermark)

CREATE TABLE IF NOT EXISTS public.sync_state (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  pipeline text NOT NULL UNIQUE,             -- e.g. 'mv_produtividade'
  last_watermark timestamptz NOT NULL,       -- last successful source_updated_at
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- Initialize once:
INSERT INTO public.sync_state (pipeline, last_watermark)
VALUES ('mv_produtividade', '2000-01-01T00:00:00Z')
ON CONFLICT (pipeline) DO NOTHING;
