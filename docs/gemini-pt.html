Estratégia de Migração: Scraper MV Saúde para API

Objetivo: Substituir o cron job em Python coletar-produtividade-mv.py por uma integração direta entre o ambiente AWS do hospital e o SaaS ParcerIA (Supabase).

1. Visão Geral da Arquitetura

A transição altera o modelo de extração via cliente (scraping) para um modelo de integração via servidor.

Estado Atual (Obsoleto)

  • Fonte: Visualizador de Relatórios HTML (http://mvpepprd.saude.go.gov.br...).
  • Método: Script Python que faz o parse de elementos DOM do HTML.
  • Risco: Alta fragilidade. Alterações na interface (UI) quebram o pipeline. A autenticação depende de cookies de sessão ou emulação de navegador.

Estado Futuro (Alvo)

  • Fonte: Ambiente AWS (Data Lake ou Warehouse gerenciado pela TI).
  • Método: Transferência de Dados Estruturados (JSON/CSV ou Replicação SQL).
  • Destino: Supabase (PostgreSQL) tabela produtividade.
  • Benefício: Estabilidade, garantia de schema, auditabilidade e segurança.

2. Definição do Esquema do Banco de Dados

Com base no arquivo produtividade - produtividade_rows.csv, a tabela de destino no Supabase requer a seguinte estrutura SQL. Certifique-se de que a equipe de TI forneça o mapeamento de dados para estas colunas.

CREATE TABLE public.produtividade (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    codigo_mv BIGINT NOT NULL,
    nome TEXT NOT NULL,
    especialidade TEXT,
    vinculo TEXT,
    
    -- Métricas (Tipos inteiros para contagens)
    procedimento INTEGER DEFAULT 0,
    parecer_solicitado INTEGER DEFAULT 0,
    parecer_realizado INTEGER DEFAULT 0,
    cirurgia_realizada INTEGER DEFAULT 0,
    prescricao INTEGER DEFAULT 0,
    evolucao INTEGER DEFAULT 0,
    urgencia INTEGER DEFAULT 0,
    ambulatorio INTEGER DEFAULT 0,
    auxiliar INTEGER DEFAULT 0,
    encaminhamento INTEGER DEFAULT 0,
    folha_objetivo_diario INTEGER DEFAULT 0,
    evolucao_diurna_cti INTEGER DEFAULT 0,
    evolucao_noturna_cti INTEGER DEFAULT 0,
    
    -- Metadados
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    data DATE NOT NULL,
    unidade_hospitalar_id BIGINT
);

-- Índices para performance em consultas comuns
CREATE INDEX idx_produtividade_data ON public.produtividade(data);
CREATE INDEX idx_produtividade_codigo_mv ON public.produtividade(codigo_mv);

3. Padrões de Integração AWS para Supabase

A equipe de TI perguntará como mover os dados do ambiente AWS deles para o Supabase. Apresente estes três padrões:

Padrão Mecanismo Prós Contras
Postgres FDW (Foreign Data Wrapper) O Supabase conecta diretamente ao AWS RDS usando a extensão postgres_fdw. Acesso em tempo real; Sem código ETL necessário; Trata tabelas AWS como tabelas locais. Requer liberação de rede (allow-listing); Performance depende da latência da rede.
AWS Lambda (Push) Uma função AWS Lambda é acionada com novos dados e chama a API do Supabase (`supabase-js`). Arquitetura desacoplada; A equipe da AWS controla o agendamento; Seguro. Requer manutenção do código da função Lambda.
Importação S3 (Batch) A TI exporta CSVs para um bucket S3. Uma Edge Function do Supabase lê e importa o arquivo. Bom para grandes atualizações em lote; Padroniza o formato. Não é tempo real; Gerenciar o estado de bloqueio/processamento de arquivos adiciona complexidade.

4. Preparação para Perguntas e Respostas Técnicas

Prepare-se para as seguintes perguntas da equipe de AWS/TI:

P1: "Como nos autenticamos no seu banco de dados?"

Resposta: "O Supabase fornece uma string de conexão PostgreSQL padrão. Podemos criar um usuário/role dedicado para o pipeline da AWS com permissões limitadas (apenas INSERT/UPDATE) na tabela produtividade para manter a segurança."

P2: "Qual é a estratégia de chave primária? Como lidamos com duplicatas?"

Resposta: "A tabela usa um UUID como chave primária. No entanto, para integridade dos dados, devemos definir uma restrição única composta (composite unique constraint), provavelmente em codigo_mv + data. Se vocês enviarem um registro que já existe, podemos usar uma estratégia de UPSERT (ON CONFLICT UPDATE) para atualizar as métricas sem criar duplicatas."

P3: "Vocês precisam de VPN ou VPC Peering?"

Resposta: "O Supabase roda na AWS (ou infraestrutura de nuvem similar dependendo da região). Podemos colocar na lista de permissões (whitelist) os endereços IP do seu NAT Gateway ou das funções Lambda. O VPC peering é possível, mas geralmente requer um plano Enterprise; a conexão padrão via SSL/TLS é geralmente suficiente para este nível de sensibilidade de dados."

P4: "Quais tipos de dados vocês esperam para as métricas?"

Resposta: "Esperamos inteiros (integers) para todos os campos baseados em contagem (ex: evolucao, prescricao). O campo data deve estar no formato ISO 8601 (AAAA-MM-DD). Valores nulos nas métricas devem ser padronizados como 0 para evitar erros de cálculo no dashboard."
